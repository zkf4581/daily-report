# 外包人员日报登记系统 — 需求与技术交接文档

> 本文档是给接手开发的工程师 / AI 的完整说明。**无需回看任何对话**，照此即可落地。
> 脚手架已搭好（见第 6 节「当前进度」），你从 **P1（数据库）** 开始即可。

---

## 1. 背景与目标

雇主找了外包，需要外包**每天汇报当天工作**。

- 诉求**不是**考勤打卡，而是**低摩擦的「每日进展同步 + 留痕」**：外包填着省事，雇主看着轻松，且历史可追溯（避免月底对不上账、方向跑偏发现太晚）。
- 目前只有 1 个外包，但**必须做成可复用的轻量多用户产品**：以后加新外包，雇主在后台点一下就能开账号，**不改代码**。

---

## 2. 角色与权限

| 角色 | 能做什么 |
|---|---|
| **管理员（雇主）** | 看**所有人**的日报；创建 / 停用 / 重置外包账号；按人筛选；看今天谁没交 |
| **外包（提交者）** | 登录后**只能看到自己**的数据；填当天日报；查看自己的历史 |

**核心安全要求**：「管理员看全部 / 外包只看自己」必须在**数据库层（RLS 行级权限）**强制执行，而不是只在前端用 if 判断。外包即使改前端、直接调接口，也**不能**读到别人的数据。

## 3. 功能需求（已确认，勿擅自改动）

### 3.1 登录
- **邮箱 + 密码**登录。
- 外包**不能自助注册**。账号由**管理员在后台创建**（填邮箱 + 初始密码，发给外包）。
- Supabase 后台需**关闭公开注册**，保证后台创建是唯一开户入口。

### 3.2 日报字段
| 字段 | 必填 | 说明 |
|---|---|---|
| 今天完成了什么 `done` | ✅ 必填 | |
| 遇到的阻塞 / 问题 `blockers` | 选填 | |
| 明天计划 `tomorrow_plan` | 选填 | |
| 耗时 `minutes_spent` | 选填 | 存分钟数，展示时转「X 小时 Y 分」 |

### 3.3 改稿规则（重要）
- 外包**当天可以反复编辑**当天的日报。
- **过了那一天就锁定**为只读历史，不能再改。
- 「今天」以**用户所在时区**判定（默认 `Asia/Shanghai`），不能用服务器或浏览器本地时钟，否则跨午夜会出错。
- 这条规则必须在**数据库层（RLS）强制**，前端只是做禁用态/隐藏编辑按钮的「体验」层。

### 3.4 管理员视图
- 默认展示**今天所有人交了没**：交了显绿、没交显红。
- 可**按人筛选**、可**选日期**查看某天。
- 点某个外包 → 看他的**历史时间线**。
- 周报 / 月报汇总属于后续阶段，本期不做。

### 3.5 邮件提醒（v1 包含）
- 每天**截止时间**（默认 **21:00**，可配置）检查谁今天没交。
- 给**未交的外包**发提醒邮件。
- 同时给**管理员**发一份「谁没交」的汇总邮件（可开关）。
- 时区统一按 `Asia/Shanghai`，所以用「每天 21:00 跑一次」的简单定时即可（外包都在国内）。
  - > 若以后有跨时区外包，再改成「每小时跑、由 Edge Function 按各人本地时间判断截止」。这是小改动，本期不做。

## 5. ⚠️ Next.js 16 关键差异（务必先读，否则会用错写法）

本项目装的是 **Next.js 16.2.9**，相比训练数据里常见的 13/14/15，有以下**破坏性变更**，写代码前必须遵守：

1. **`middleware.ts` → 改名为 `proxy.ts`**
   - 文件放项目根目录，导出的函数名也叫 `proxy`（不是 `middleware`）。
   ```ts
   // proxy.ts
   import { NextResponse, type NextRequest } from 'next/server'
   export async function proxy(request: NextRequest) { /* ... */ }
   export const config = { matcher: ['/admin/:path*', '/today', '/history', '/api/admin/:path*'] }
   ```

2. **`cookies()` / `headers()` 现在是异步的**，必须 `await`：
   ```ts
   import { cookies } from 'next/headers'
   const cookieStore = await cookies()
   ```

3. **`params` / `searchParams` 是 Promise**，必须 `await`：
   ```ts
   // app/admin/people/[id]/page.tsx
   export default async function Page({ params }: { params: Promise<{ id: string }> }) {
     const { id } = await params
   }
   ```

4. **Route Handler 的 params 同样是 Promise**：
   ```ts
   export async function PATCH(req: NextRequest, { params }: { params: Promise<{ id: string }> }) {
     const { id } = await params
   }
   ```

5. **`fetch` 默认不再缓存**（v13 是默认缓存）。需要缓存时显式写 `cache: 'force-cache'` 或 `next: { revalidate }`。本项目数据全走 Supabase，注意 Supabase 查询要保证拿到实时数据。

6. **Turbopack 是默认**（`next dev` / `next build` 默认用它），无需 `--turbopack` 标志。

7. **数据变更**用 Server Action（`'use server'`）+ `revalidatePath()` / `redirect()`。

## 7. 数据库设计（P1）

迁移文件放 `supabase/migrations/0001_init.sql`。

### 7.1 `profiles`（与 Supabase 自带 `auth.users` 一对一）
| 列 | 类型 | 约束 |
|---|---|---|
| `id` | uuid | 主键，外键 → `auth.users(id)` ON DELETE CASCADE |
| `email` | text | NOT NULL（冗余一份便于管理员列表/筛选） |
| `full_name` | text | 显示名，可空 |
| `role` | text | NOT NULL DEFAULT `'contractor'`，`CHECK (role IN ('admin','contractor'))` |
| `is_active` | boolean | NOT NULL DEFAULT `true`（停用 = 禁止登录） |
| `timezone` | text | NOT NULL DEFAULT `'Asia/Shanghai'` |
| `created_at` | timestamptz | DEFAULT `now()` |

- 在 `auth.users` 上加 `AFTER INSERT` 触发器，自动建对应 `profiles` 行：复制 `email`，`role` 取自 `raw_app_meta_data->>'role'`（缺省 `contractor`）。

### 7.2 `reports`
| 列 | 类型 | 约束 |
|---|---|---|
| `id` | uuid | 主键 DEFAULT `gen_random_uuid()` |
| `user_id` | uuid | NOT NULL，外键 → `profiles(id)` ON DELETE CASCADE |
| `report_date` | date | NOT NULL，这份日报对应的「本地日期」 |
| `done` | text | NOT NULL |
| `blockers` | text | 可空 |
| `tomorrow_plan` | text | 可空 |
| `minutes_spent` | integer | 可空，`CHECK (minutes_spent >= 0)` |
| `created_at` | timestamptz | DEFAULT `now()` |
| `updated_at` | timestamptz | DEFAULT `now()`，更新时由触发器刷新 |

- **唯一约束** `UNIQUE (user_id, report_date)`：每人每天一份；提交 = 按此键 **upsert**。
- 索引：`(report_date)`、`(user_id, report_date DESC)`。

### 7.3 `app_settings`（单行配置表）
- 至少含：`reminder_hour`（int，默认 21）、`notify_admin_digest`（bool，默认 true）。
- 用单行（可用固定主键 `id = 1` + CHECK 限制只能一行）。改配置不用重新部署。

### 7.4 `reminder_log`（邮件幂等，P6 用）
- `UNIQUE (user_id, report_date)`，发过提醒就插一行，防止定时任务重复跑导致重复发信。

### 7.5 `is_admin()` 辅助函数 + RLS

```sql
create or replace function public.is_admin()
returns boolean language sql security definer stable
set search_path = public as $$
  select exists (
    select 1 from public.profiles
    where id = auth.uid() and role = 'admin' and is_active
  );
$$;
```
> `SECURITY DEFINER` 让该函数读 `profiles` 时**不再触发 profiles 的 RLS**，避免「profiles 策略里查 profiles」的无限递归经典坑。

**开启两张表的 RLS，策略如下：**

- **profiles**
  - `SELECT`：`auth.uid() = id OR public.is_admin()`（本人或管理员）
  - `UPDATE`：`auth.uid() = id`（本人改自己名字/时区；管理员改 role/is_active 走 service_role 路由，自动绕过 RLS）
  - `INSERT` / `DELETE`：普通用户无策略（由触发器和 service_role 路由处理）

- **reports**
  - `SELECT`：`user_id = auth.uid() OR public.is_admin()`
  - `INSERT`：`user_id = auth.uid() AND report_date = (now() AT TIME ZONE (select timezone from public.profiles where id = auth.uid()))::date`
  - `UPDATE`：`USING` 和 `WITH CHECK` **都用上面这条** `report_date = 本人时区今天` → 这就是「当天可改、过天锁定」的**真正强制点**。
  - `DELETE`：无（不允许删）

> `service_role` 自动绕过所有 RLS，所以管理员创建账号的服务端路由**不需要**额外策略。

### 7.6 区分管理员的方式
- 权威来源是 **`profiles.role`**（RLS 只信这个）。
- 创建账号时**同时**把 `role` 写进 `auth.users` 的 `app_metadata.role`，仅作为 proxy 中间件快速判断用（中间件读 metadata 省一次查询），敏感操作仍以 `profiles.role` 为准。

## 8. Supabase 客户端 helper（P1 尾 / P2 头）

按 `@supabase/ssr` + **Next 16 异步 cookies** 写，建议建：

- `lib/supabase/client.ts` — 浏览器端：`createBrowserClient(URL, ANON)`。
- `lib/supabase/server.ts` — 服务端（Server Component / Server Action）：`const cookieStore = await cookies()`，再 `createServerClient(URL, ANON, { cookies: { getAll, setAll } })`。`setAll` 用 try/catch 包住（在纯 RSC 里 set 会抛错，忽略即可）。
- `lib/supabase/admin.ts` — **仅服务端**：用 `@supabase/supabase-js` 的 `createClient(URL, SERVICE_ROLE, { auth: { persistSession:false, autoRefreshToken:false } })`。**只准在 Route Handler / Server Action 里 import**，绝不能进客户端 bundle。
- `lib/supabase/proxy-session.ts` — 给 `proxy.ts` 用的会话刷新 helper（用 `request.cookies` 读、把刷新后的 cookie 写回 `NextResponse`）。

环境变量见根目录 `.env.local.example`（需复制成 `.env.local` 填真值）：
```
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
SUPABASE_SERVICE_ROLE_KEY=         # 仅服务端，勿加 NEXT_PUBLIC_
RESEND_API_KEY=                    # P6
REMINDER_FROM_EMAIL=               # P6，已在 Resend 验证的发信地址
REMINDER_AUTH_TOKEN=               # P6，cron 与 Edge Function 的共享密钥
```

## 9. 页面 / 路由

| 路由 | 角色 | 用途 | 主要 shadcn 组件 |
|---|---|---|---|
| `/` | 公开 | 落地页 → 登录入口 | Magic UI hero |
| `/login` | 公开 | 邮箱密码登录 | Input, Button, Card, Sonner |
| `/today` | 外包 | 填 / 改当天日报 + 锁定态 | Textarea, Card, Badge；Magic UI 提交成功动画 |
| `/history` | 外包 | 自己的历史（只读） | Table, Card, Badge, Accordion |
| `/admin` | 管理员 | 今天所有人交了没（缺交标红）+ 按人筛 + 选日期 | Table, Badge（绿/红）, DatePicker, DropdownMenu |
| `/admin/people/[id]` | 管理员 | 某外包历史时间线 | Table, Card, Tabs, Badge |
| `/admin/users` | 管理员 | 创建 / 停用 / 重置外包账号 | Dialog, Switch, AlertDialog |

**路由守卫（`proxy.ts`）**：未登录 → `/login`；外包访问 `/admin*` → `/today`；管理员落地 → `/admin`。中间件读 `app_metadata.role` 提速；服务端组件对敏感操作再核验 `profiles.role`。

**数据访问**：Server Components / Server Actions 用绑定 cookie 的 SSR 客户端「以用户身份」读写，RLS 自动授权——管理员查到全部、外包只查到自己（查询里仍建议带 `where user_id` 以利用索引、表达清晰）。

### 9.1 管理员创建外包账号（安全做法）
`service_role` 密钥**绝不进浏览器**。走服务端路由 `POST /api/admin/users`：
1. 读会话，确认已登录；
2. 以调用者身份核验是管理员（查 `is_admin()` 或 `profiles.role`），否则 403；
3. 用 `lib/supabase/admin.ts` 的 service_role 客户端调
   `auth.admin.createUser({ email, password, email_confirm: true, app_metadata: { role: 'contractor' } })`
   （`email_confirm:true` 跳过确认邮件，管理员直接设初始密码）；
4. 触发器自动建 `profiles` 行，需要的话再用 admin 客户端补 `full_name` / `timezone`。

- **停用 / 启用**：`PATCH /api/admin/users/[id]`（管理员限定）→ 改 `profiles.is_active`，并 `auth.admin.updateUserById(id, { ban_duration: '876000h' | 'none' })` 让停用账号**真的登不进来**。
- **重置密码**：`auth.admin.updateUserById(id, { password })`。
- ⚠️ 用 `auth.admin.createUser()`，**不要**用 `signUp()`（后者会用 service 客户端换成用户 JWT，是错的）。

## 10. 邮件提醒机制（P6）

链路：`pg_cron`（定时）→ `pg_net`（发 HTTP）→ Edge Function `send-reminders` → 查出今天没交的人 → Resend 发邮件。

1. 启用扩展：`create extension if not exists pg_cron; create extension if not exists pg_net;`
2. 把 `project_url` 和 `REMINDER_AUTH_TOKEN` 存进 Supabase Vault。
3. **定时任务**：`0 21 * * *`（每天 21:00，单时区）；`net.http_post` 带 `Authorization: Bearer <REMINDER_AUTH_TOKEN>` 调 `…/functions/v1/send-reminders`。
4. **Edge Function `send-reminders`**（Deno）：
   - 先**校验** `Authorization` 头里的 token（Edge Function 公网可达，必须校验）；
   - 用自身环境里的 service_role 查：所有 `is_active` 的外包中，今天（`now() AT TIME ZONE profiles.timezone` 的日期）**没有** `reports` 行的人；
   - 逐个用 Resend 发提醒（`RESEND_API_KEY` + 已验证发信域名 `REMINDER_FROM_EMAIL`）；
   - 若 `app_settings.notify_admin_digest` 为真，给管理员发「谁没交」汇总；
   - **幂等**：每发一个就往 `reminder_log (user_id, report_date)` 插一行（唯一约束防重复发）。

文件放 `supabase/functions/send-reminders/index.ts`。

## 14. 安全红线（务必遵守）

- `SUPABASE_SERVICE_ROLE_KEY` **绝不**加 `NEXT_PUBLIC_` 前缀，**绝不**在客户端组件里 import `lib/supabase/admin.ts`。
- 权限判断**以数据库 RLS 为准**，前端控制只是体验，不是安全边界。
- 所有写操作在系统边界做输入校验（邮箱格式、必填项、`minutes_spent >= 0`）。
- Edge Function 必须校验 `REMINDER_AUTH_TOKEN`（公网可达）。
- 错误信息不要泄露敏感细节（如「该邮箱不存在」要避免被用于探测用户）。
